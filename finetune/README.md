# Dictum reformatter — fine-tune pipeline

Purpose-built model for the spoken→agent-ready reformat task. bf16 LoRA on
Qwen2.5-Instruct (Apache-2.0), native Windows + HF TRL. Full plan:
`docs/PLAN-0.3-finetune.md`. Runs on the RTX 5090 (32 GB) — no WSL2, no
bitsandbytes, no Unsloth.

## 0. Setup (order matters — this is the Blackwell gotcha)

```powershell
cd finetune
py -3.12 -m venv .venv ; .\.venv\Scripts\Activate.ps1
python -m pip install -U pip

# 1) torch FIRST, with the cu128 wheel (CUDA 12.8+ is mandatory for sm_120):
pip install torch --index-url https://download.pytorch.org/whl/cu128

# 2) VERIFY before anything else — must print (12, 0) and not error:
python -c "import torch; print(torch.cuda.get_device_capability()); print((torch.randn(8,8,device='cuda')@torch.randn(8,8,device='cuda')).sum().item())"

# 3) the rest:
pip install -r requirements.txt
```

If step 2 errors with a cuBLAS failure on the matmul, reinstall torch with the
**cu130** wheel instead (`.../whl/cu130`) — a known Blackwell cu128 issue.
(`cu129` does not exist.) These pins churn; if something breaks, paste the error.

## 1. Data

`data/train.jsonl` + `data/eval.jsonl` — chat-format pairs, generated by the
`dictum-reformat-dataset` workflow (Claude writes them here). Each line:
```json
{"messages":[{"role":"system","content":"..."},{"role":"user","content":"<messy>"},{"role":"assistant","content":"<clean>"}]}
```

## 2. Train

```powershell
# 3B — the GPU SKU:
python train.py --model Qwen/Qwen2.5-3B-Instruct   --out out/reformat-3b-al   --epochs 2 --all-linear
# 1.5B — the CPU-fallback SKU:
python train.py --model Qwen/Qwen2.5-1.5B-Instruct --out out/reformat-1.5b-al --epochs 3 --lr 1.5e-4 --all-linear
```
`--all-linear` is the default recipe as of 2026-07-22: on dataset v2 it beat
attention-only for BOTH sizes (smoke-gate fails/188: 3B 7→4, 1.5B 13→5).
~15–45 min each on the 5090. **Watch the `[mask-check]` line** — it must print
only the clean completion, never the messy prompt. It keeps the best checkpoint
(eval_loss), not the last. If it under-cleans (leaves fillers), re-run with
`--all-linear`; if it memorizes/over-edits, drop `--rank 8`.

## 3. Export to GGUF (after training — I'll add `export.py`; manual flow for now)

1. Merge adapter → fp16 HF model (`peft merge_and_unload` / `save_pretrained_merged`).
2. `python llama.cpp/convert_hf_to_gguf.py merged --outfile m-f16.gguf --outtype f16`
3. *(recommended)* `llama-imatrix` over a few hundred real `messy→clean` samples (in-domain).
4. `llama-quantize m-f16.gguf m-Q4_K_M.gguf Q4_K_M` (+ imatrix).
- The trained chat template/system prompt **must** be in `tokenizer_config.json` at
  convert time; verify `chat_template` + `eos_token_id` (`<|im_end|>`=151645) in the
  GGUF metadata; run llama.cpp with `--jinja`. This mismatch is the #1 export failure.
- If Q4_K_M fails eval, step up Q5_K_M / Q6_K.

## 4. Eval (after export — I'll add `eval.py`)

Two layers on the **Q4_K_M GGUF** (not the HF model), temp 0, base-vs-FT:
- **Layer 1 — deterministic gates** (= the product guardrail chain): identifier-presence,
  length-ratio, negation-preserved, not-answered. A gate fail is a fail regardless of judge.
- **Layer 2 — LLM-judge rubric** on Layer-1 passes: structure-decision / question-preserved /
  not-answered / identifiers-intact / meaning-preserved. Pairwise A/B, swap-and-average for
  position bias. Report per-category win/tie/loss vs base.
